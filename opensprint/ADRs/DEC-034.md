## Question

DEC-029 established Docker-compose as the integration test recipe and the deployment shape. That works locally and reflects production. But running `docker compose up --wait --build` in GitHub Actions on every PR adds 1–3 minutes of image-build time per run — significant friction for CI feedback. How should the e2e suite run in CI?

## Operator Decision

Run the same e2e test bodies in CI by bringing the stack up as **native Node child processes**: `apps/coinbase-mock/dist/main.js` and `apps/hub/dist/main.js` are spawned as processes pointing at each other via `localhost` ports. The test fixtures (`lifecycle.test.ts`) are unchanged; only the bringup helper changes.

The test runner picks the bringup mechanism via an `INTEGRATION_BRINGUP` env var:

- `INTEGRATION_BRINGUP=docker` — the existing `composeUp`/`composeDown` path (DEC-029).
- `INTEGRATION_BRINGUP=process` — the new native-process path.

Default behavior: in a CI environment (the `CI` env var is truthy — set by GitHub Actions and most CI providers), default to `process`. Otherwise default to `docker`. The `test:ci-e2e` package script sets `INTEGRATION_BRINGUP=process` explicitly so operators can also run the CI path locally for diagnosis.

DEC-029's Docker recipe **remains the canonical local-dev and production-deployment shape**. The CI path is an additional bringup option, not a replacement.

## Consideration Factors

- **Native Node processes (chosen)** — fast (~5s vs 60s+ for Docker bringup), no image registry to maintain, uses the same `dist/main.js` artifacts that prod consumes. Cost: a small refactor of `helpers.ts` to support two bringup modes; CI doesn't exercise the Dockerfile build itself (caught only by the `docker` path locally).
- **Docker in CI** — exact same shape as local + production, fully validates the deployment recipe in CI. Cost: 1–3 min/run is real friction at PR cadence; `docker buildx` cache helps but is ceremony to maintain.
- **GitHub Actions `services:`** — declarative container bringup. Cost: still container bringup (same speed problem); locks the workflow to GitHub's specific syntax; we already own the orchestration in `helpers.ts`.
- **In-process composition** (spawn hub + mock in the same Node process via NestJS test factories) — fastest of all. Rejected: loses the network-isolation property — bugs around port binding, JSON serialization on the wire, real WS ws-frame handling, and HTTP request lifecycle would be invisible. The whole point of e2e is to validate the *real* network surfaces.
- **Pre-built test image in GHCR** — push a hub+mock image once per main commit, pull it for PR runs. Rejected: adds release infra (registry creds, push step, image pruning) for a take-home with no production fleet to feed.

## Rationale

The headline insight is that DEC-029's framing — "the compose recipe IS the deployment shape" — is a *deployment* claim, not a *test runtime* claim. The integration tests don't need to use the same bringup mechanism as production for every run; they need the same protocol surfaces and the same fault-injection knobs against the same mock fixtures. Native process bringup gives us those at CI speed without invalidating DEC-029.

The Docker path keeps catching deployment-shape regressions (Dockerfile changes, runtime image layout, env propagation) on local dev runs and on any PR that touches the Dockerfile or compose. CI runs the process path on every PR for fast feedback. Each path catches what it's good at.

The auto-detection (`CI` env truthy → process) is the operator's "intuitive default" framing: developers running locally without setting any flags get Docker (production-shape) by default; CI runs without setting any flags get processes (fast). The explicit `test:ci-e2e` script lets either audience switch paths without arguments.

## Invalidation Trigger

Revisit if:
- The Docker bringup catches a class of regression that the process path misses repeatedly — would suggest making the Docker path also part of CI (perhaps on main-branch pushes only, not every PR).
- CI runtime budgets become non-issues (very fast runners, generous minutes) — could promote the Docker path to be the only CI path.
- A class of test emerges that genuinely requires the deployment shape (e.g. tests of network policies, reverse-proxy headers, Docker healthcheck behavior). Those would explicitly use `INTEGRATION_BRINGUP=docker`.
- The hub gains a non-trivial init step that's only exercised inside the Docker image (e.g. distroless-specific behavior). Then process bringup diverges from production and the safety claim weakens.
