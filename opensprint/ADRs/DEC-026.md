## Question

How does the dashboard fetch data — should it use the existing production data plane, or expose new dashboard-specific endpoints?

## Operator Decision

The dashboard uses the **same data plane the production eval will exercise**:

- **HTTP poll on `/status`** (every 1–2 seconds) — slow-moving aggregate state: uptime, mode, upstream connection state, per-topic freshness, consumer counts.
- **WS gateway subscribe** to one or more `market://...` URIs — live book updates routed through the same fan-out path any production consumer uses.

No new endpoints are introduced. The dashboard validates by demonstration that the production data plane is sufficient for human operability.

## Consideration Factors

- **HTTP poll + WS subscribe (chosen)** — exercises production surfaces; double-counts as proof that the data plane is operator-friendly. Polling cadence on `/status` is bounded and cheap.
- **WS-only push for everything** — would need a `status.subscribe` channel on the gateway; couples the gateway protocol to dashboard concerns. `/status` is slow-moving; polling is simpler and bounded.
- **Server-Sent Events for telemetry** — adds a third transport with no clear benefit over HTTP poll. Splits the surface.
- **New dashboard-specific REST/GraphQL endpoint** — defeats the entire point: the dashboard becomes a special citizen, and production observability tools would have to use a different surface.

## Rationale

DS-LLM-USABILITY says the MCP surface should be sufficient for an agent to succeed first-try. DS-OPERATOR-USABILITY's parallel claim is that `/status` + WS gateway should be sufficient for a human operator to see the system clearly. By building the dashboard on these surfaces, we *prove* that claim — not just assert it.

Polling cadence (1–2s) is fast enough for the slow-moving telemetry state but not so fast that it dominates the hub's load profile. Live book updates take the WS path because they are inherently push-shaped and the gateway already does fan-out.

## Invalidation Trigger

- Dashboard needs information not exposed via `/status` or WS gateway. Resolution: extend `/status` (preferred) or add a new field, but keep the surface count constant.
- Polling cadence proves too coarse for an emerging UX requirement (e.g. sub-second connection-count updates). Could promote relevant fields to push-based emission via WS gateway.
- A third audience (e.g. external integrators, alerting systems) emerges with different shape needs and pushes for a dashboard-specific API.
