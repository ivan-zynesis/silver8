## Question

How do we structure the codebase as a monorepo, and what tooling manages dependencies and builds?

## Operator Decision

pnpm workspaces + Turborepo. Package layout:

```
packages/
├── core/                # interfaces: Bus, OrderBookStore, Registry, types
├── core-memory/         # InMemoryBus, InMemoryOrderBookStore, InMemoryRegistry
├── ingestion/           # Coinbase adapter, sequence/gap, book maintenance
├── gateway-ws/          # WS server, subscribe protocol
├── mcp-server/          # MCP server: tools + resources
└── observability/       # logger, status surface, metrics

apps/
└── hub/                 # composition root: NestJS modules wired per MODE
```

## Consideration Factors

- **pnpm + Turborepo** — pnpm has the best workspace semantics in npm-land (strict by default, fast); Turbo gives task graph + caching. Well-trodden, low risk.
- **Bun workspaces** — would have been the choice if we picked Bun; Bun's task runner is sufficient at this size.
- **Nx** — more powerful but heavier configuration; overkill for ~6 packages.
- **Single package** — rejected; the package boundary is the architectural seam (`core` interfaces vs `core-memory` impl is the literal demonstration of the adapter pattern).

## Rationale

The package graph mirrors the architectural seams (DEC-004). `core` exports interfaces; `core-memory` is one impl; future `core-network` (Redis/NATS) drops in alongside without anything in `core` changing. Reading the package graph teaches the architecture — strong eval signal.

`pnpm` over npm/yarn for: strict module resolution (no phantom deps), dramatically smaller node_modules, and faster installs. Turbo for: task caching across packages, parallel builds, dependency-aware test ordering.

## Invalidation Trigger

Revisit if:
- We adopt Bun runtime (DEC-001 reversed) — Bun workspaces would replace.
- Build complexity outgrows Turbo (very unlikely at this scale).
