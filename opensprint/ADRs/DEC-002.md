## Question

How do we compose the application's components — modules, dependency injection, lifecycle ordering — at production-grade quality from day 1?

## Operator Decision

NestJS as the composition framework, with the Fastify HTTP adapter for the HTTP shell.

## Consideration Factors

- **Plain TypeScript with hand-wired composition root** — lightest. Composition is ~50 lines of `new` calls in `main.ts`. Wins on legibility for small systems. Loses on: hierarchical module composition, dependency-graph topological start/stop ordering, scoped configuration, scoped logging, encapsulated sub-system testing. As the operator noted, this is "the demo version."
- **`tsyringe` (or InversifyJS)** — DI container only. Better than hand-wired for DI ergonomics, but doesn't solve module hierarchy or lifecycle ordering. Middle ground that doesn't fully address the requirement.
- **NestJS with Fastify adapter** — modules-of-modules, lifecycle hooks (`OnModuleInit`, `OnApplicationBootstrap`, `OnApplicationShutdown`) run in topological dependency order, scoped DI, `Test.createTestingModule().overrideProvider()` for sub-tree testing. Non-HTTP components (Coinbase WS client, `ws` server, MCP SDK) live as plain providers; HTTP shell uses Nest's `@Controller` over Fastify. Familiar to many engineers.
- **Effect (with Layer system)** — the most principled alternative: typed end-to-end dependency graphs, managed resources (acquire/release), structured concurrency, Layer composition is essentially CDK-for-application-runtime. Conceptually closest to what the operator described as desired ("hierarchical, dependency graph properly managed"). Rejected for this 2-week deliverable due to: steeper learning curve, uneven reviewer familiarity (eval criterion includes "how easily for another engineer to understand and extend"), and a comparatively novel idiom for a team that may inherit this code. Worth revisiting for a system whose long-term maintainers are Effect-fluent.

## Rationale

The operator named hierarchical composition + dependency-graph management + lifecycle ordering as day-1 requirements (production-correct stance, not deferrable). Once those are firm, NestJS earns its weight: the conventions *are* the architecture. The Fastify adapter avoids the historical Express baggage and gives better HTTP performance.

Critically: NestJS owns *DI and lifecycle*. We do not use Nest's opinionated WebSocket gateway (Socket.io-first, wrong for our protocol) — `WsServer` is a plain provider wrapping `ws`. Same for the MCP server: a plain provider wrapping `@modelcontextprotocol/sdk`. This isolates Nest to where it adds value.

## Invalidation Trigger

Revisit if:
- The team becomes Effect-fluent and wants the typed Layer-based composition.
- NestJS overhead becomes a measurable problem (unlikely at this scale).
- We need to extract `core` to a non-Nest consumer; this works today because `core` interfaces have no Nest dependency.
