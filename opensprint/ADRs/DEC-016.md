## Question

We're shipping a monolith for the take-home but designing for split-tier deployment. How do we make the split-tier capability *real* and verifiable, not just claimed in the architecture write-up?

## Operator Decision

The single binary supports three deployment variants, selected at startup by the `MODE` environment variable. The composition root in `apps/hub/main.ts` imports a different combination of NestJS modules per variant:

```
MODE=monolith   (default â€” dev, demo, CI)
   CoreMemoryModule + IngestionModule + GatewayWsModule + McpServerModule
   â†’ in-memory Bus / OrderBookStore / Registry; one process; fully operational

MODE=ingestion  (production ingestion tier)
   CoreNetworkModule + IngestionModule
   â†’ bus/store/registry adapters point at NATS/Redis;
     no consumer-facing surfaces

MODE=gateway    (production gateway tier)
   CoreNetworkModule + GatewayWsModule + McpServerModule
   â†’ bus/store/registry adapters point at NATS/Redis;
     no upstream Coinbase WS
```

`CoreNetworkModule` (Redis/NATS-backed adapter implementations) is **deferred** â€” we don't build it as part of v1. But the composition entry points for `ingestion` and `gateway` modes ship: starting the binary in those modes without `CoreNetworkModule` available will fail with a clean configuration error message ("no Bus adapter configured for MODE=gateway; CoreNetworkModule is required").

## Consideration Factors

- **Three variants, network module deferred** â€” operationalizes the seam (DEC-004): the binary itself enforces that ingestion and gateway communicate only via Bus/OrderBookStore/Registry, because the wrong module isn't even imported in tier modes. Strong eval signal.
- **Single `monolith` mode only** â€” simpler; but the seams stay aspirational. A reviewer can't *verify* the architecture supports split-tier.
- **Three variants, network module shipped** â€” most ambitious; adds substantial scope (Redis adapter for three seams + integration testing) for no marginal eval value beyond demonstrating the variants exist.

## Rationale

By making `MODE=gateway` and `MODE=ingestion` real composition entry points (even if they fail without a network module), we *prove* the seams are honest. The package graph shows `gateway-ws` doesn't import `ingestion`; the composition shows the binary can be assembled without it. This is the production-correctness commitment the operator asked for: the architecture is real at day 1, not retrofitted.

The deferred `CoreNetworkModule` is named in the architecture write-up with the swap target ("Redis pub/sub for Bus, Redis hash for OrderBookStore, Redis sorted-set for Registry; or NATS for Bus") â€” a future maintainer knows exactly what to build.

## Invalidation Trigger

Revisit if:
- The eval reviewer pushes back on shipping non-functional modes (`MODE=gateway` without a working network module) â€” could either ship `CoreNetworkModule` or drop the modes from the binary.
- A fourth tier emerges (e.g., a separate "control plane" mode for status/admin); the variant selection should expand cleanly.
