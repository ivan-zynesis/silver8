## Question

Coinbase L2 updates carry sequence numbers, and gaps indicate dropped messages. How does the hub detect, signal, and recover from gaps?

## Operator Decision

The Coinbase adapter tracks per-product sequence numbers on every L2 update. On gap detection:

1. **Mark the topic stale**: the `OrderBookStore` enters a stale state for that symbol; downstream snapshots are tagged `stale: true` until resync completes.
2. **Emit stale signaling**: subscribers (Bus subscribers, MCP resource subscribers, WS clients) receive an explicit stale notification. WS: `{event: "stale", resource: "market://coinbase/book/BTC-USD", reason: "sequence_gap"}`. MCP: a `notifications/resources/updated` followed by resource content marked stale.
3. **Trigger resync**: the adapter unsubscribes and re-subscribes to the L2 channel for the affected product, receives a fresh snapshot, and rebuilds book state.
4. **Resume normal flow**: once the new snapshot lands and the adapter is processing fresh updates against it, stale flag clears; subscribers receive a fresh snapshot delivery.

Stale detection is also driven by heartbeats (DS-COINBASE-WS provides them): if no update or heartbeat arrives for a configured threshold, the topic is marked stale even without an explicit gap.

## Consideration Factors

- **Detect, signal, automatically resync** — correct under stress; consumers always know whether their view is current. This is what DS-BRIEF "correctness under stress" is asking for.
- **Detect and signal but require manual resync** — useless for an LLM consumer who has no ops dashboard.
- **Ignore gaps** — wrong; book diverges from Coinbase reality silently.
- **Restart the entire WS connection on any gap** — overcorrection; affects all symbols on that connection.

## Rationale

Sequence-gap handling is the single most important correctness mechanism in the ingestion subsystem. The eval criterion explicitly names "correctness under stress: reconnections and stale states" — this ADR is its primary fulfilment. Per-symbol resync (rather than full-connection restart) is the targeted fix that doesn't penalize neighboring symbols.

The stale signal is also visible in the MCP tool surface (`get_top_of_book` returns a `stale: boolean` field) and `/docs` failure-modes section, so an LLM agent encountering it has a documented recovery path.

## Invalidation Trigger

Revisit if:
- Coinbase changes its sequence semantics or introduces a new resync mechanism (e.g., gap-fill API).
- Per-symbol resync proves too noisy in practice (e.g., gaps cluster across symbols on the same connection); could promote to connection-level reconnect.
