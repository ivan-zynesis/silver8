## Question

Should the consumer connection registry hold socket references in process memory, or in some external store (Redis, database, etc.)?

## Operator Decision

In-memory only — and this is not a tradeoff, it's a structural fact. Socket references **cannot** live anywhere except the process holding the file descriptor. A `ws.Socket` is not serializable; the send capability is process-local.

External stores (Redis, Postgres) can hold *metadata about* connections but cannot replace the in-memory map. They become useful only when the system goes split-tier and Redis aggregates demand across processes — and even then, sockets stay in the process they connected to.

## Consideration Factors

| Approach | Holds socket ref? | What it actually buys |
|---|---|---|
| In-memory `Map` | ✅ yes (only it can) | Direct, fast, simple |
| Redis (KV) | ❌ no — sockets aren't serializable | Cross-process visibility of subscriptions; only useful in split-tier |
| Postgres | ❌ no — category error | Persisted metadata, but useless because sockets are gone on restart anyway |
| Sticky LB + per-node maps | ✅ on the node it landed on | Multi-node deploys with sockets pinned per node — same as in-memory within a node |
| Erlang/Elixir process-per-conn | ✅ beautifully | Not relevant; we picked Node |

## Named Hazards

In-memory holds real failure modes. We name them and prevent each by construction:

1. **Unbounded send queue** — slow consumer; we keep `ws.send()`-ing; OOM. Fix: bounded per-consumer ring buffer in our code (DEC-011); check `socket.bufferedAmount` against a watermark.
2. **Listener leaks** — `socket.on(...)` never `off()`-ed on disconnect, pinning closures. Fix: every `on()` paired with `off()` through a single cleanup function.
3. **Map-that-grows-only** — consumer IDs lingering in Registry indices after disconnect. Fix: one disconnect path; every cleanup goes through `Registry.removeConsumer(id)`.
4. **Hot-symbol fanout amplification** — N consumers on BTC-USD; one upstream tick becomes N sends. Fix: serialize the outgoing message once per upstream tick, then `socket.send(buffer)` per consumer.
5. **Heap fragmentation under churn** — operational, not a leak; `--max-old-space-size` with headroom + container memory limit + RSS monitoring.
6. **What's lost on restart** — all sockets, queues, and book state. Documented per DS-BRIEF; clients reconnect, books rebuild from Coinbase snapshot.

## Invariants Enforced By Tests

- **Churn test**: 10k connect/disconnect cycles; RSS stable within a defined envelope.
- **Slow-consumer test**: a consumer that never reads has its queue bounded and is eventually disconnected; other consumers unaffected.
- **Subscribe/unsubscribe storm**: no orphan entries in any Registry index after the storm settles.

## Rationale

There is no real alternative for the part of the problem we're solving (holding live sockets). The architectural commitment is therefore to *manage the hazards by design*, not to pretend an external store solves them.

## Invalidation Trigger

Revisit if:
- We adopt a runtime that changes the socket-locality assumption (Cloudflare Durable Objects, Erlang-style process model).
- The named hazards prove insufficient — i.e., a memory issue in production traces to a hazard not on this list.
