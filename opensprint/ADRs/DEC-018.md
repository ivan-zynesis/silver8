## Question

If/when the gateway tier autoscales, what metric drives the autoscaler decision?

## Operator Decision

**Active consumer connection count per pod**, exposed as a Prometheus gauge `hub_active_consumer_connections{pod=...}`. Target around 1000 connections/pod; tunable.

The metric is consumed via the standard custom-metrics path:

- Kubernetes: Prometheus → `prometheus-adapter` → HPA custom metric → `targetAverageValue: 1000`.
- ECS/Cloud Run: emit to CloudWatch / Cloud Monitoring; target tracking on that metric.
- Fly.io: native connection-count autoscaling.

L4 LB (NLB / GCP TCP LB) with **least-connections** policy distributes new TCP connections; once a connection lands, WS is naturally sticky.

## Consideration Factors

| Metric | Why right or wrong |
|---|---|
| **Active WS connections (chosen)** | Direct, predictable, easy to reason about; tracks the actual cost driver for a fanout gateway. |
| CPU% | Misleading for WS — a pod with 5000 idle subscribers is near 0% CPU; a pod with 500 noisy ones runs hot. CPU correlates with *message rate*, not connection load. |
| Memory | Closer to right, but noisy from heap fragmentation and transient allocations. |
| Connection rate (conns/sec) | Useful as a *leading* indicator (autoscaler reacts before saturation); not a primary signal. |

## Rationale

For a fanout WebSocket service, connection count is the natural unit of capacity. CPU and memory are downstream effects that vary with consumer behavior; connections are the directly-controllable leading indicator. This is the canonical metric for this class of system.

## Invalidation Trigger

Revisit if:
- A specific symbol or consumer profile causes per-connection costs to vary dramatically (e.g., 99% of consumers subscribe to BTC-USD, so connection count understates message-fanout cost). At that point a composite metric (`connections × subscribed_topics_message_rate`) might be needed.
- We adopt a runtime where connection count isn't directly observable (e.g., serverless connection-as-Lambda-invocation models — but DEC-024 rules these out).
