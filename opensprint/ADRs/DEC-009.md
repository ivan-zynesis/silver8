## Question

The hub's only subscribable topic in v1 is L2 order book per trading pair (e.g., BTC-USD). Coinbase delivers an initial snapshot followed by incremental updates. Where does the assembled book state live, and what does the hub expose?

## Operator Decision

The hub maintains the L2 book as state. Coinbase snapshots and updates flow into the `OrderBookStore`, which applies them to produce the current book. **What the hub exposes via MCP resources and WS gateway is a top-N levels snapshot, not raw deltas.** Diff application stays inside the hub; consumers (especially LLM agents) never need to apply diffs.

The single topic in v1 is `market://coinbase/book/<symbol>` — full L2 state of one trading pair. Adding `trades.*` or `ticker.*` is a future extension that requires no architecture change (just adapter event handling and resource registration); they are explicitly out of scope for v1.

## Consideration Factors

- **Maintain book in hub; expose snapshots** — LLM-friendly (no diff application required). Costs: real state subsystem inside the hub with gap-recovery, stale-detection, etc. This *is* the hard part of the system.
- **Forward raw deltas** — simpler to implement; consumers (even LLMs) must apply diffs. Defeats DS-LLM-USABILITY.
- **Maintain book + expose deltas only** — sophisticated consumers might want this, but our headline consumer is an LLM that should not be in the diff-applying business.

## Rationale

DS-LLM-USABILITY demands that the agent can ask "what's the current top of book for BTC-USD?" and get a usable answer immediately, not a stream of deltas. DS-COINBASE-WS forces us to assemble book state from snapshot+updates anyway. Putting that assembly inside the hub aligns the natural shape of the upstream data with the natural shape of the LLM-facing API.

For v1: `book` only. The architecture supports trades/ticker as cheap additions when needed; they bring no new structural complexity, so deferring loses nothing.

## Invalidation Trigger

Revisit if:
- A consumer class emerges that genuinely wants raw deltas (e.g., an internal trading system replicating book state for arbitrage). At that point, expose deltas as a separate topic kind, don't change the headline.
- Book maintenance correctness is harder to prove than expected; consider third-party order-book libraries.
