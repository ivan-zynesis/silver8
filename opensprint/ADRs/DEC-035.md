## Question

DEC-013 commits the hub to streaming via the protocol-native `resources/subscribe` flow, and DEC-014 names HTTP+SSE as the primary transport. The current implementation runs HTTP in stateless mode (`sessionIdGenerator: undefined`, fresh `McpServer` + transport per request) — a workaround for the SDK's one-transport-per-server constraint that surfaced as "Already connected to a transport" errors when a singleton `McpServer` was shared across requests. The workaround keeps `tools/call` working but cannot honor `resources/subscribe`: there is no long-lived transport over which `notifications/resources/updated` can flow. How should HTTP transport be structured so the original DEC-013 commitment is honored?

## Operator Decision

Run HTTP MCP transport in **stateful sessions**. The hub maintains a session map keyed by the SDK's `Mcp-Session-Id` header (issued on `initialize`, echoed on every subsequent request). Each session owns its own `McpServer` + `StreamableHTTPServerTransport` pair, with `resources: { subscribe: true }` re-enabled. The transport is long-lived for the session's duration; `notifications/resources/updated` flows over its SSE channel as bus events arrive.

```
sessions: Map<sessionId, {
  server:          McpServer,           // configured per-session, subscribe: true
  transport:       Transport,           // one connect() at session birth, never reconnected
  subscribedUris:  Set<ResourceURI>,    // tracked per-session for fan-out
  busOff:          Unsubscribe[],       // bus subscriptions wired into the session's transport
}>
```

Discipline (the rules whose violation produced the earlier "Already connected" bug):

1. **One server per session.** Sessions never share.
2. **`server.connect()` exactly once per server.** On the request that creates the session (the `initialize` body).
3. **Every subsequent request for that session** calls `transport.handleRequest(req, res, body)` on the existing connected transport — never `connect()` again.

Session lifecycle:

- Created when the request has no `Mcp-Session-Id` header AND the JSON-RPC body is `initialize`. The SDK's `sessionIdGenerator: () => randomUUID()` mints the id; `onsessioninitialized` registers the session in the map.
- Looked up by `Mcp-Session-Id` for any other request.
- Reaped on transport `close` (client closes the SSE stream OR DELETEs `/mcp` with the session id) OR on a 5-minute idle timer (`MCP_SESSION_IDLE_MS`, default 300_000) — matches the upstream-socket idle posture in DEC-027 so an operator's mental model is "5 min of nothing → cleanup."
- SIGTERM drain (DEC-019 path) iterates sessions, sends `notifications/silver8/rebalance` to each, waits the deadline, force-closes stragglers.

Stateless per-request mode is **deleted**. The `createPerRequestServer()` factory and the matching branch in `McpController` are removed; stateful covers every prior use case (smoke tests just need to thread the session id between calls, which is one-line standard MCP client behavior).

`/status.consumers.mcp` reflects the live session count; per-topic `consumerCount` aggregates correctly across MCP sessions and WS gateway consumers (the dashboard already exposes the column — DS-OPERATOR-USABILITY).

No concurrent-session cap in monolith mode (one operator, demo scope). A future split-tier deployment running `MODE=gateway` with `CoreNetworkModule` would set a cap when the cost of a session × replica count starts mattering; explicitly out of scope here.

## Consideration Factors

- **Stateful (chosen)** — restores DEC-013's protocol-native streaming over the DEC-014 primary transport. Session map costs ~few hundred bytes + N bus listeners per concurrent agent; bounded by idle reaper. Standard MCP HTTP server pattern (mirrors the SDK's documented stateful example).
- **Keep stateless + direct streaming consumers to the WS gateway** — pragmatic but defeats the point of having MCP as a first-class surface. Agents would need to know about a second protocol. Defeats DS-LLM-USABILITY's "first-try success" criterion.
- **Per-request server with bus-state-replay-on-poll** — agents poll `get_book_snapshot` repeatedly. Workable but not MCP-native; loses the push semantics that `resources/subscribe` exists to provide; defeats DEC-013.
- **Singleton server with custom transport multiplexer** — the SDK fights us on this; would require forking. Out of scope.
- **Keep stateless mode alongside stateful** — adds a code path with no use case (every stateless flow works fine in stateful with one extra header). Dead code is debt; ship the canonical pattern, delete the workaround.

## Rationale

The stateless mode that landed in commit df290b4 was a fix-commit-level workaround, not an architectural commitment. The framing "stateless is a temporary degradation of DEC-013 until stateful is implemented" was implicit in the comment block of `mcp.controller.ts` ("HTTP clients that want streaming should connect via the stdio bridge or use the WS gateway") but was never recorded as a deferral. This ADR closes the loop by formalizing the upgrade and, in doing so, finishes what DEC-013 started.

The "1 server per session, `connect()` once, `handleRequest` many" discipline is what the SDK's contract requires. Stateful mode honors all three rules naturally because each session owns its pair for life. The earlier singleton bug violated rule 1 (sharing) which forced violating rule 2 (reconnecting); stateless avoided rules 1 and 2 by forfeiting rule 3 (transport doesn't survive the request). Stateful is the only design that holds all three.

`/status.consumers.mcp` becoming session-aware is the visible-behavior side of the change: agents subscribed via MCP show up in the same column as WS consumers, and the dashboard's "active topics" table aggregates correctly. The accounting was always there in the WS gateway; this ADR makes the MCP side equally legible.

## Invalidation Trigger

Revisit if:
- A class of agent emerges that genuinely cannot track session ids (would be unusual; mcp-remote, the official SDK, MCP Inspector, and Claude Desktop all do). Resolution: re-introduce a stateless fallback path, but as a degraded mode that explicitly disables `subscribe`.
- Concurrent-session memory becomes a real limit (e.g. `MODE=gateway` shedding load across replicas, with thousands of agents per pod). Resolution: cap sessions per pod, surface backpressure as `Mcp-Session-Id` rejection on `initialize`. Sibling decision to DEC-018's autoscale signal.
- The SDK changes its session-handling API in a backward-incompatible way. Resolution: migrate the session map shape; the high-level discipline (one server per session, one connect, many handleRequests) survives the change.
- A request shape emerges that genuinely cannot fit into a session (e.g. a webhook delivery to `/mcp` from a third-party that refuses to follow the MCP handshake). Resolution: route those to a different endpoint, not `/mcp`.
