## Question

DS-BRIEF declares `/docs` a first-class deliverable, written for an LLM consumer. What is the minimum structure and content?

## Operator Decision

`/docs` ships with the following structure, treated with the same rigor as code:

```
docs/
├── 00-overview.md            Purpose, scope, non-goals; what this hub is and isn't.
├── 01-getting-started.md     Connect via MCP (HTTP+SSE and stdio); first invocations; setup.
├── 02-mcp-tool-reference.md  Per-tool: name, when to use, args, returns, examples, errors.
├── 03-mcp-resources.md       Resource URI scheme, resources/subscribe lifecycle, content shape.
├── 04-topics.md              Per-topic: name, schema, update cadence, real example payload.
├── 05-worked-examples.md     "Agent wants the current mid for BTC-USD: call X with Y, expect Z."
├── 06-failure-modes.md       Stale, lagged, rebalance, unknown symbol — what the agent sees and does.
├── 07-ws-gateway.md          The non-MCP consumer surface: subscribe protocol, op/event reference.
└── 08-architecture.md        One-page architecture write-up (links into ADRs for depth).
```

**Writing rules:**

- Plain language up front; jargon defined or avoided.
- Every example is real (executed against a running hub, output captured).
- Every failure mode names the exact event/notification the agent will see and the recommended response.
- Tool reference cross-links to topic schemas and examples.
- No "TODO" placeholders shipped — incomplete docs are worse than missing docs.

## Consideration Factors

- **Treat docs as a first-class deliverable** — the brief explicitly says so, and the eval criterion ("how far a fresh agent gets when handed only your documents") is measured directly against this. Worth dedicated time, not afterthought polish.
- **README + auto-generated reference** — auto-gen helps with sync but auto-generated docs read like reference material, not teaching material. Worked examples and failure modes can't be auto-generated.
- **Defer to inline tool descriptions only** — minimal but unhelpful for an agent that wants context before it starts calling tools.

## Rationale

DS-LLM-USABILITY is unusual as a quality attribute precisely because it makes documentation load-bearing. The eval criterion translates directly to "did the docs work?" — so we structure the docs to *be* worked, not just read. Worked examples and failure modes are the highest-signal sections; we prioritize those over comprehensive reference.

A practical test: when the docs are draft-ready, hand them to a fresh LLM agent (with no source code access) and have it attempt the worked examples. Anywhere it stumbles, the docs need work. This is the canonical fix loop.

## Invalidation Trigger

Revisit if:
- Tool reference outgrows readability (consider per-tool pages).
- Worked examples diverge from current behavior (need a CI hook to keep them in sync).
- A new audience (e.g., venue integrators adding a second exchange) needs different docs structure (would add a `09-extending-venues.md`, not restructure).
