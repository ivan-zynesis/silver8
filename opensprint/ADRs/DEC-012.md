## Question

What protocol do non-MCP consumers use to subscribe to topics over the gateway?

## Operator Decision

WebSocket with a small JSON op protocol. Client → server messages:

```json
{ "op": "subscribe",   "resource": "market://coinbase/book/BTC-USD" }
{ "op": "unsubscribe", "resource": "market://coinbase/book/BTC-USD" }
{ "op": "ping" }
```

Server → client messages:

```json
{ "event": "snapshot",  "resource": "...", "data": { ... }, "stale": false }
{ "event": "update",    "resource": "...", "data": { ... } }
{ "event": "stale",     "resource": "...", "reason": "sequence_gap" }
{ "event": "lagged",    "resource": "...", "dropped": 142 }
{ "event": "rebalance", "reason": "scale-down", "deadline_ms": 30000 }
{ "event": "error",     "code": "...", "message": "..." }
```

Protocol uses the same `market://` URIs as MCP (DEC-008) so docs and examples are unified.

## Consideration Factors

- **WebSocket with small JSON ops** — bidirectional, simple, every language has clients. The canonical answer for consumer-facing pub/sub.
- **SSE + POST** — one-way for receive, separate channel for subscribe. More moving parts; awkward.
- **GraphQL subscriptions** — over-engineered for the take-home; schema/codegen/resolver layer is too much surface for a single subscribe API.
- **gRPC streaming** — strong typing but requires client codegen; less LLM-friendly than JSON over WS.

## Rationale

The gateway exists to demonstrate that the hub serves more than one consumer surface — proving the architecture isn't accidentally MCP-only. WS with JSON is the simplest possible "real" protocol: trivially testable from a browser console, an `wscat` CLI, or any HTTP client with WS support. The op/event shape mirrors the MCP `resources/subscribe` flow conceptually, so an engineer reading both at once sees the parallel.

## Invalidation Trigger

Revisit if:
- A specific consumer class needs strong typing (gRPC) or schema introspection (GraphQL).
- Protocol grows past a handful of ops/events; might warrant a more formal schema.
