## Question

DS-BRIEF requires a status surface (active connections, uptime per connection, message rates). How do we expose it, given consumers include both engineers (HTTP) and LLM agents (MCP)?

## Operator Decision

Two surfaces, **same payload**:

- **HTTP `GET /status`** — JSON response, intended for engineers, dashboards, and automated checks.
- **MCP `get_hub_status()` tool** — same JSON response, exposed as an MCP tool so LLM agents can read it without leaving the MCP context.

Payload includes:

```json
{
  "uptime_seconds": 12345,
  "upstream": {
    "coinbase": {
      "connection_status": "connected" | "reconnecting" | "disconnected",
      "connected_since": "2026-05-02T12:00:00Z",
      "subscribed_channels": [...],
      "messages_per_sec": 234,
      "last_message_at": "..."
    }
  },
  "topics": [
    {
      "uri": "market://coinbase/book/BTC-USD",
      "consumer_count": 7,
      "messages_per_sec": 142,
      "stale": false,
      "last_update_at": "..."
    }
  ],
  "consumers": {
    "ws_gateway": { "connected": 23, "active_subscriptions": 41 },
    "mcp_server":  { "connected": 5,  "active_subscriptions": 12 }
  }
}
```

## Consideration Factors

- **Two surfaces, same payload** — consumers (engineers and LLMs) get parallel access; one source of truth for the data shape.
- **HTTP only** — engineer-friendly but invisible to LLM agents driving the system, who'd have to leave MCP to check status.
- **MCP only** — LLM-friendly but awkward for monitoring tools and dashboards; everyone expects an HTTP endpoint.
- **Different payloads for each surface** — drift; documentation diverges; one source of truth lost.

## Rationale

DS-LLM-USABILITY explicitly wants the LLM consumer to be a first-class user of the system; status is part of operating the system, so it must be visible from inside MCP. DS-BRIEF requires a status surface generally; HTTP `/status` is the canonical engineer-facing form. Sharing the payload keeps the docs and tests honest.

## Invalidation Trigger

Revisit if:
- The payload grows too large for an MCP tool response (would need pagination or filtering).
- Status surfaces diverge in audience needs (e.g., engineers want raw counters, LLMs want narrative summary). Could expose two views over the same underlying state.
