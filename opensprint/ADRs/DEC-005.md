## Question

What semantics must the in-memory `Bus` implementation honor so that it is genuinely swap-equivalent with a future distributed implementation (NATS, Redis pub/sub, etc.)?

## Operator Decision

The in-memory `Bus` is **not** a synchronous EventEmitter wrapper. It honors the same semantics as a distributed bus from day 1:

- **Async publish**: `publish()` returns `Promise<void>` and resolves after dispatch is queued — never propagates subscriber errors back to the publisher.
- **Microtask-queued delivery**: subscribers are invoked on the next microtask, not inline on the publisher's stack.
- **Lossy semantics**: no durability, no replay; subscribers falling behind drop messages from their bounded queue (DEC-011).
- **No cross-topic ordering**: only FIFO *within* a topic. Subscribers must not depend on cross-topic ordering.
- **Demand observability**: `onDemandChange(topic, count)` lets the ingestion tier drive upstream subscriptions from refcounted demand.

## Consideration Factors

- **Honoring distributed semantics** — slightly more code in the in-memory impl; testing exercises the real semantics; "drop-in replacement" claim is honest.
- **Synchronous EventEmitter wrapper** — simplest and fastest. But code that depends on synchronous delivery, in-stack error propagation, or cross-topic ordering will break the moment we swap to NATS. The seam becomes theatrical.

## Rationale

The danger of an adapter pattern is that the default implementation has different semantics from the production one, making "drop-in replacement" a fiction. We avoid this by committing to the harder semantics from day 1. The in-memory `Bus` becomes a true reference implementation of the contract, not a degenerate special case.

This is testable: a Bus semantics test suite (publish-then-subscribe-late drops the early message; bounded subscriber queue overflows correctly; slow subscriber does not block publisher) lives in `core-memory` and would also be the test harness for any future `core-nats` or `core-redis`.

## Invalidation Trigger

Revisit if:
- A distributed Bus implementation actually ships and we discover its semantics are stricter than what we encoded (e.g., total ordering across topics, durable replay) — adjust the in-memory impl up to match.
- Performance profiling shows the microtask-queued delivery is a bottleneck (very unlikely at our scale).
