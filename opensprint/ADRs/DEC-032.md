## Question

How do existing surfaces — HTTP `/status`, MCP `list_topics()`, MCP `describe_topic(uri)`, and helpful errors — accommodate the catalog/active split introduced in DEC-030?

## Operator Decision

`/status` payload evolution:

```diff
{
  "uptime_seconds": 12345,
  "upstream": { ... },
- "topics": [
-   { "uri": "...", "consumer_count": 7, "messages_per_sec": 142,
-     "stale": false, "last_update_at": "..." }
- ],
+ "catalog": [
+   { "uri": "market://coinbase/book/BTC-USD",
+     "venue": "coinbase", "kind": "book", "symbol": "BTC-USD",
+     "description": "..." }
+ ],
+ "active": [
+   { "uri": "...", "consumer_count": 7, "messages_per_sec": 142,
+     "stale": false, "last_update_at": "..." }
+ ],
  "consumers": { ... }
}
```

The previous `topics` field is **renamed to `active`** (no migration cost — nothing depends on the old name in production). `catalog` is added.

MCP surface evolution:
- `list_topics()` returns the catalog. Authoritative answer to "what can I ask for?" — empty `active` no longer means empty `list_topics()`.
- `describe_topic(uri)` accepts any catalog URI, including those with zero current consumers, and returns the static descriptor from the adapter's catalog.
- `get_hub_status()` returns the same payload as `/status`, including the new split.

Helpful-error sourcing: WS gateway and MCP both source the "available URIs" enumeration from the union of all venue adapters' catalogs.

## Consideration Factors

- **Rename `topics` → `active`; add `catalog` (chosen)** — the names match the concepts; readers stop being confused; one source of truth.
- **Keep `topics` and add `catalog`** — backward-compat-shaped, but there is no production yet to be compatible with. Leaves a misleading field name behind.
- **Single `topics` field with a `state` discriminator (`"available" | "active"`)** — collapses the split syntactically; loses the readability win that was the whole point.
- **Two MCP tools (`list_catalog`, `list_active`)** — duplicates the surface. `list_topics()` should keep its name; "topics" semantically means the catalog from a consumer's perspective.

## Rationale

The conceptual split deserves visible naming. Calling the active list `active` and the catalog `catalog` puts a fresh reader (human or LLM) one read away from understanding the system; calling them `topics` and `available_topics` muddies that. With no installed base, the rename is free.

Augmenting `list_topics()` to ground in catalog (rather than introducing a new MCP tool) preserves the LLM-facing surface DS-LLM-USABILITY established. An agent that called `list_topics()` in v1 against a populated hub gets the same answer in v2 against a cold hub — that is the point.

DEC-022 and DEC-015 are augmented, not invalidated; both remain active.

## Invalidation Trigger

Revisit if:
- Catalog grows large enough that the `/status` payload becomes unwieldy — would split `/status` (active state) from a separate `/catalog` endpoint and add MCP-side pagination.
- Active and catalog diverge in unexpected ways (e.g. an active topic that is no longer in the catalog because the catalog refreshed and a pair was removed mid-flight). Need to define what `active`-not-in-catalog means; suggest treating it as a transient state with explicit signaling.
- An audience emerges that wants only the catalog (e.g. an external symbol-listing service). Could split into `/catalog` and `/status` endpoints, but the MCP surface stays consolidated.
