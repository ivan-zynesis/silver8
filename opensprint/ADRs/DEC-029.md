## Question

How do we run end-to-end tests covering the complete data-plane lifecycle (consumer connects → upstream attaches → snapshot → gap recovery → consumer disconnects → upstream releases → drain) and how does this relate to deployment?

## Operator Decision

`apps/integration-tests` is a **vitest-driven test package** that orchestrates a multi-service `docker-compose.yml`:

```
services:
  coinbase-mock:   replays captured Coinbase sessions (DEC-028)
  hub:             built from existing Dockerfile, configured to point at coinbase-mock
```

Tests boot the compose stack, connect WS / MCP clients to the hub, drive the lifecycle via fault-injection commands sent to the mock, and assert the hub's behavior end-to-end.

Test shapes the harness must cover:

- Consumer subscribes → upstream channel attaches → snapshot delivered.
- Inject sequence gap → topic goes stale → resync triggers → fresh snapshot delivered → topic recovers.
- Last consumer disconnects → channel unsubs immediately; socket closes after configured idle window (DEC-027).
- Slow consumer → bounded queue overflows → `lagged` event → sustained-overflow disconnect.
- SIGTERM → rebalance hint broadcast → consumers reconnect → drain completes.

**Crucially**, this same `docker-compose.yml` IS the deployment recipe. Production replaces `coinbase-mock` with the real venue and replaces `docker compose` with the production orchestrator (Fly Machines, Cloud Run, Kubernetes). Same image, same env vars, same service topology. **No separate IaC artifact is needed for the take-home** — the compose file is the IaC, and the production extension is an orchestration swap, not a new artifact.

## Consideration Factors

- **Vitest + docker-compose (chosen)** — explicit, debuggable, no extra dependency. Compose syntax doubles as deployment shape documentation.
- **testcontainers library** — programmatic container lifecycle from the test process. More idiomatic but adds a dependency and obscures the compose structure that we want to double as deployment recipe.
- **Long-running shared test environment** — flaky, hard to assert deterministically against, defeats the "operator can run it locally" property in DS-OPERATOR-USABILITY.
- **Pure unit tests with mocks (no docker)** — already what we have for component-level correctness. Doesn't catch lifecycle interaction bugs across services. Necessary but not sufficient.
- **Separate IaC milestone** — explicitly rejected by the operator's reframe: the docker-compose recipe is the deployment shape; orchestrator/registry swaps are operational extensions, not new artifacts.

## Rationale

Dockerized e2e tests give us deterministic CI assertions for behavior that crosses service boundaries — exactly the surface that DEC-027's demand-driven lifecycle and DEC-010's gap recovery need to be tested against. Component-level tests (already extensive — 89 in v1) verify each piece in isolation; the integration suite verifies the system as composed.

The deployment-recipe insight is structural: in a take-home that explicitly defers IaC (DEC-024), the docker-compose orchestrating tests *is* the closest thing to deployment infrastructure we have. Treating it as both a test harness and a deployment artifact preserves a single source of truth. Production extension becomes obvious — replace mock with real, replace compose with orchestrator — rather than introducing a separate IaC universe.

This ADR extends DEC-024: the production deployment shape is no longer documented only in prose; it's documented in code (`docker-compose.yml`), and the gap to production is named explicitly in the architecture write-up.

## Invalidation Trigger

- Hub grows into actual split-tier deployment requiring `CoreNetworkModule` (Bus / Registry / Store backends). The compose file would need to add Redis/NATS services; the ADR's "compose-as-recipe" property still holds.
- CI runtime cost of docker-based tests becomes prohibitive (e.g. minutes per run on every push). Could split tests into a fast unit suite and a slow e2e suite that runs on a schedule or on label-triggered PRs.
- A third-party WS testing harness emerges that's clearly better than vitest + compose for our shape. Unlikely; vitest + compose is the boring default.
