## Question

What is the naming scheme for subscribable topics, and how does it map to MCP resource URIs?

## Operator Decision

Topics are addressed by URI: `market://<venue>/book/<symbol>`. Examples:

- `market://coinbase/book/BTC-USD`
- `market://coinbase/book/ETH-USD`

This URI is used uniformly:
- As the topic key in the `Bus`.
- As the MCP resource URI for `resources/subscribe`.
- As the subscribe target in the WS gateway protocol.
- In `/docs` examples and error messages.

## Consideration Factors

- **Venue-prefixed URI (`market://coinbase/book/BTC-USD`)** — multi-venue ready; reads naturally to humans and LLMs; `book` segment leaves room for `trades`, `ticker` later (DEC-009 keeps L2 only for v1, but the scheme accommodates expansion).
- **Flat name (`book.BTC-USD`)** — simpler today; would need to be redesigned the moment a second venue arrives. Punts the inevitable.
- **Coinbase-shaped name (`level2.BTC-USD`)** — leaks venue terminology; defeats the normalization in DEC-007.

## Rationale

DS-LLM-USABILITY rewards URI schemes that read naturally to an LLM. `market://` reads as a domain (market data), `coinbase` is the venue, `book` is the data kind, `BTC-USD` is the instrument. An agent can construct these URIs from documentation alone with high reliability — exactly the first-attempt-success criterion.

The scheme also gives us safe expansion paths without breaking changes:
- `market://coinbase/trades/BTC-USD` (future)
- `market://binance/book/BTC-USDT` (future)

## Invalidation Trigger

Revisit if:
- A topic shape doesn't fit the `<venue>/<kind>/<symbol>` pattern (e.g., cross-venue derived data; aggregated indices).
- LLM agents struggle with the scheme in practice (would be visible in `/docs` testing).
