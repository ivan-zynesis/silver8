## Question

How does catalog availability interact with the readiness contract that DEC-020 established (`/readyz` only true after upstream connect + book initialized)?

## Operator Decision

The `VenueAdapter` exposes `catalogReady: boolean`. The hub's `/readyz` gates on the conjunction of:

- All registered venue adapters report `catalogReady === true`, AND
- Existing readiness criteria from DEC-020 (upstream-state-aware where applicable).

For the v1 hardcoded-pairs catalog (DEC-031), `catalogReady` flips to `true` synchronously at startup — the catalog lives in code, no I/O required. For a future REST-discovery adapter, `catalogReady` would gate on first successful product-fetch, with retry/backoff behavior owned by the adapter.

## Consideration Factors

- **Gate `/readyz` on catalog (chosen)** — downstream consumers get a guarantee: by the time `/readyz` is true, `list_topics()` returns a meaningful list and helpful errors can enumerate alternatives. Pre-shapes the REST-discovery upgrade by making the readiness contract independent of source.
- **Don't gate on catalog; treat empty catalog as a normal state** — simpler, but loses the property that a "ready" hub can answer the LLM-facing questions DS-LLM-USABILITY cares about. A cold REST-discovery adapter could announce ready before catalog populates, and an agent calling `list_topics()` would see an empty list and conclude (correctly per the empty list, wrongly per intent) that nothing is available.
- **Have catalog readiness gate `/healthz` instead of `/readyz`** — wrong semantic. Liveness ("is the process alive?") is independent of catalog state; readiness ("can I serve traffic correctly?") is what catalog gates.

## Rationale

The promise DEC-030 made — that catalog is authoritative for "what can I ask for?" — only holds if the system does not accept traffic before it can keep that promise. Readiness gating is the enforcement mechanism. Tying it to a per-adapter property keeps the gate independent of catalog source: hardcoded today is trivially ready, REST-discovery tomorrow is gated on first fetch, hybrid after that gates on the allowlist (immediate) plus best-effort metadata hydration (configurable).

Splitting this from DEC-031 (rather than folding it in) is deliberate. DEC-031 records the *current source*; DEC-033 records the *readiness contract*. They have different invalidation triggers — DEC-031 changes when we ship REST discovery; DEC-033 changes only if the readiness model itself needs reshaping.

## Invalidation Trigger

Revisit if:
- A catalog-source choice emerges where readiness can't be cleanly defined (e.g. eventually-consistent catalog from a stream of venue events). Would need a "minimum viable catalog" threshold rather than a binary ready flag.
- A hub mode emerges that should serve some traffic before catalog is available (maintenance status endpoints, etc.). Would require splitting readiness into multiple readiness probes for different traffic classes — cleaner than weakening this gate.
- Cross-venue policy (the deferred ALL-vs-ANY question) demands that the hub be ready when *any* adapter is ready (not all). Would change the conjunction in this ADR.
