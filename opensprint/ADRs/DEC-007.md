## Question

How do we structure the upstream ingestion code so that adding a second venue (Binance, Kraken, etc.) is "a simple exercise" as DS-BRIEF requires?

## Operator Decision

Define a `VenueAdapter` interface in `core/`. Each venue implements it. Internally, all messages are translated to a normalized format before they hit the Bus or OrderBookStore.

```typescript
interface VenueAdapter {
  readonly venue: string                    // "coinbase"
  start(): Promise<void>
  stop(): Promise<void>
  subscribe(symbol: string, channel: ChannelKind): Promise<void>
  unsubscribe(symbol: string, channel: ChannelKind): Promise<void>
  health(): VenueHealth
  readonly events: EventEmitter             // emits NormalizedMessage
}
```

The Coinbase adapter parses Coinbase-shaped messages (with sequence numbers, snapshot/update format, side strings, etc.) into the normalized format. A future Binance adapter does the same translation from Binance shapes.

## Consideration Factors

- **Adapter pattern with normalized format** — components downstream of the adapter (OrderBookStore, Bus, Gateway, MCP) are venue-agnostic. Adding a venue is one new package.
- **Bake Coinbase-specific shapes throughout** — fastest for v1. But the requirement explicitly is "make adding venues a simple exercise"; this fails it.
- **Adapter pattern but no normalization (forward raw)** — middle ground; downstream still has to know venue shapes. Pushes the translation cost out into every consumer.

## Rationale

DS-BRIEF makes multi-venue extensibility a first-class architectural requirement. Even though we ship single-venue v1, the only way to honor the requirement is to bake the boundary in now. Normalization at the adapter is the right place because the adapter is already doing parse-from-wire work.

The normalized format includes: venue, symbol, channel, sequence (where applicable), payload, server timestamp.

## Invalidation Trigger

Revisit if:
- A second venue is added and the normalization proves insufficient (some venue concept doesn't fit) — extend the format, but keep the boundary.
- The eval feedback indicates the adapter pattern is over-engineered for single-venue scope (unlikely given the explicit DS-BRIEF requirement).
