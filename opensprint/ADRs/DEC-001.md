## Question

What language and runtime should we use to build the hub?

## Operator Decision

TypeScript on Node.js.

## Consideration Factors

- **TypeScript on Node** — mature MCP SDK (`@modelcontextprotocol/sdk`), excellent WebSocket ecosystem (`ws`), boring stable runtime, large hiring pool.
- **TypeScript on Bun** — same MCP SDK; cleaner single-binary deploy via `bun build --compile`; faster startup. Tradeoff: younger ecosystem, less production-validated.
- **Python** — official MCP SDK exists; pydantic schemas are nice; `asyncio` is solid. Tradeoff: WS performance is OK but not Node-tier; heavier deployment story.
- **Go** — best WS server performance; channels are a natural fanout primitive; single-binary native. Tradeoff: MCP SDK is much less mature; Zod-equivalent schema-to-tool flow doesn't exist; more friction to write the LLM-facing tool surface.

## Rationale

Stability is the chosen optimization. Node's MCP SDK is the most mature option and pairs natively with Zod, which we'll use for schema-to-tool wiring (DEC-015). TypeScript gives us static type safety across the registry, bus, and adapter interfaces. WS server ergonomics are excellent. Two-week deadline doesn't reward exotic runtime choices.

Bun was the close second; rejected for stability over single-binary panache, given we already plan a Docker image deliverable (DEC-024 area).

## Invalidation Trigger

Revisit if:
- MCP SDK support for Node deteriorates relative to other languages.
- A specific performance bottleneck (per-connection event-loop saturation) cannot be addressed via vertical scale or sharding.
- Single-binary deployment becomes a hard requirement and Docker becomes unacceptable.
