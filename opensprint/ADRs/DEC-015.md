## Question

What tools does the MCP server expose, and what are the design rules for naming, argument schemas, and descriptions?

## Operator Decision

Initial tool surface:

| Tool | Purpose |
|---|---|
| `list_topics()` | Returns array of `{ uri, kind, venue, symbol, description }` for every available topic. The starting point for any agent. |
| `describe_topic(uri)` | Returns schema, update cadence, example payload, and freshness metadata for a topic. The agent's "what does this give me?" lookup. |
| `get_top_of_book(symbol, venue?)` | Point-in-time snapshot: best bid, best ask, mid, spread, timestamps, `stale: boolean`. The most common query. |
| `get_book_snapshot(symbol, venue?, depth?)` | Top-N levels of the order book as a snapshot. For agents that need more than top-of-book. |
| `get_hub_status()` | Active connections, uptime, message rates per topic, upstream connection health. The same payload as the HTTP `/status` (DEC-022). |

Plus resources at `market://<venue>/book/<symbol>` for streaming via `resources/subscribe` (DEC-013).

**Design rules:**

1. **Action-verb names**: `get_top_of_book`, not `gtob` or `topOfBook`. Verbs in the name communicate intent.
2. **Strongly typed arguments via Zod**: enums where the value space is closed (`venue: z.enum(["coinbase"])`, `symbol: z.enum([...])`). Zod schemas flow directly into MCP tool argument schemas; one source of truth.
3. **Descriptions explain *when* to use the tool, not just what it does.** Example: "Use this to get the current best bid and best ask for a trading pair. Returns a snapshot tagged with staleness — if `stale: true`, the upstream feed has gapped and a resync is in progress; the values are the last known good snapshot."
4. **Errors are actionable**: `unknown symbol BTC-USDT; available symbols: BTC-USD, ETH-USD, SOL-USD` rather than `400 Bad Request`.

## Consideration Factors

- **This surface (5 tools + resources)** — covers the four MCP capabilities required by DS-BRIEF (list, describe, subscribe-and-stream, snapshot query) plus the status surface required by the brief's connection registry section.
- **Single mega-tool with op param** — fashionable in some MCP designs (`market_data({op: "get_top_of_book", ...})`); rejected because it makes argument typing weaker and the LLM has to deconstruct the op-routing logic.
- **More tools (per-symbol, per-venue)** — proliferation; doesn't help an LLM that's better with parameterized tools.

## Rationale

DS-LLM-USABILITY makes this the most user-visible decision in the system — a fresh agent reads tool names + descriptions + argument schemas + `/docs` and decides whether the system is usable. The surface has been chosen to map directly onto the questions an agent would ask: what's available (`list`), what's it look like (`describe`), what's the current state (`get_*`), and how do I subscribe (resources).

Zod-driven typing means the schema published to the MCP client is automatically derived from the same definition we validate with — no drift.

## Invalidation Trigger

Revisit if:
- LLM testing reveals confusion between `get_top_of_book` and `get_book_snapshot` (consider merging with depth defaulting).
- Adding venues or topic kinds requires reshaping arguments rather than adding values to enums (would be a smell).
- A tool is repeatedly misused by agents in a way the description didn't anticipate (rewrite the description; this is the canonical fix).
