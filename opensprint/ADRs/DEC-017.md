## Question

What's the realistic scaling path for this system as load grows? What scales horizontally, what scales vertically, and what is the role of refcount in scaling?

## Operator Decision

**Refcount is correctness/efficiency, not scaling.** It decides whether to subscribe upstream at all (drop the Coinbase sub when no consumer wants it), keeping us under per-connection limits. It does not help handle 10× the consumer load.

Scaling levers, by what's growing:

| What's growing | How we scale | Notes |
|---|---|---|
| **Consumer count** (more WS / MCP clients) | Add edge/gateway replicas; sticky-session L4 LB; new connections distribute by least-connections. | The horizontal lever. Each gateway subscribes via Bus to whatever its consumers want; bus copies the same message to N gateways cheaply. |
| **Symbol count** (more upstream subs per Coinbase) | Shard ingestion tier by symbol: node A owns BTC, ETH; node B owns SOL, AVAX. Leader election + shard assignment. | Deferred — only needed when one box can't hold all the books. Not v1. |
| **Volume per symbol** (BTC-USD goes wild) | Vertical scale of ingestion. | Each symbol's book is single-source-of-truth; one core handles BTC-USD comfortably. |

**Symbol sharding is operational placement, not a partition protocol** (see DEC-021). It's "which node owns which symbol's upstream WS" — closer to consistent-hashing than to Kafka partitions.

## Consideration Factors

- **Edge horizontal + ingestion vertical-then-sharded** — matches the natural shape: consumers grow faster than symbols; ingestion stays small.
- **Horizontal-replicate the entire monolith** — described and rejected during exploration: each replica opens its own Coinbase WS, multiplying upstream load and creating book-state divergence. The architecture's central refcount promise is violated.
- **Symbol-partition the messaging layer** — described and rejected: that's reimplementing Kafka inside our bus.

## Rationale

The eval criterion "scalability... attention" is satisfied by *naming the scale path correctly*, not by building it. The operator surfaced the conceptual mix-up (refcount vs scaling) explicitly during exploration; the ADR records the resolution so future maintainers don't relearn it.

This ADR is the parent of the autoscale signal (DEC-018), drain semantics (DEC-019), and the production deployment shape (DEC-024).

## Invalidation Trigger

Revisit if:
- A scale shape we didn't anticipate emerges (e.g., one symbol with truly extreme volume that needs intra-symbol sharding — would require splitting the order book by price band, which is non-trivial).
- The refcount-as-correctness framing is challenged by a load pattern where it does provide a scaling effect (would be informative).
