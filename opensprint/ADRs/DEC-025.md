## Question

How do we expose the hub's live state and data to a human operator — for the take-home demo and as the foundation of the eventual ops dashboard mentioned in the brief's status-surface requirement?

## Operator Decision

Build a real frontend in `apps/dashboard` using **Vite + React + TypeScript**. The hub serves it statically at `/dashboard` from the same container that runs the data plane. MVP scope today — status panel showing the `/status` payload plus a single live book ticker for one selected symbol — but the foundation is production-realistic and grows into the full ops dashboard without rewriting.

```
apps/
├── hub/         (existing — adds static asset serving for /dashboard/*)
└── dashboard/   (new — Vite + React + TS)
```

## Consideration Factors

- **Real Vite + React app (chosen)** — production-realistic; same stack any team would use for an ops dashboard. Take-home walkthrough goes from "watch this curl output" to "watch the book tick in the UI." Same container, no new tier.
- **Throwaway HTML + vanilla JS** — minimal effort, but the operator's framing was explicit: "mini for demo but same foundation as the actual dashboard." Throwaway doesn't satisfy "foundation."
- **Separate static-host service** — operationally cleaner in a large team but over-engineered for v1; adds a tier purely for static asset serving.
- **Next.js / SSR framework** — overkill for a dashboard whose data is fetched client-side anyway; SSR adds build complexity for no reader-facing benefit.

## Rationale

DS-OPERATOR-USABILITY positions the dashboard as a first-class surface, parallel to MCP tools for LLM consumers. The take-home eval criteria reward "structured to support... [agents] and engineers building and operating effectively" — the dashboard is the operator-facing half of that.

Bundling with the hub (single container) is consistent with DEC-016 monolith mode and the docker-compose-as-deployment-recipe framing in DEC-029. A separate dashboard tier introduces deployment complexity that production doesn't yet need.

This ADR is the *foundation* commitment; the data plane choice (HTTP + WS) is DEC-026.

## Invalidation Trigger

- Dashboard outgrows what makes sense to bundle with the hub (e.g. needs auth, multi-tenant rendering, server-side templating).
- A separate frontend team takes over and wants its own deployment surface.
- SSR or dynamic rendering becomes a hard requirement (very unlikely for an ops dashboard).
