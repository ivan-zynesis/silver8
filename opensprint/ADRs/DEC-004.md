## Question

How do we structure the system so the monolith we ship today can become a split-tier deployment (separated ingestion + gateway with a real fanout bus) tomorrow, without rewriting the components?

## Operator Decision

Define three named seams as interfaces in `core/`, with `core-memory` shipping in-memory implementations. All component code (ingestion, gateway, MCP) depends only on the interfaces, never on a concrete implementation.

```typescript
interface Bus {
  publish(topic: string, msg: NormalizedMessage): Promise<void>
  subscribe(topic: string, handler: Handler): Unsubscribe
  onDemandChange(cb: (topic: string, count: number) => void): Unsubscribe
}

interface OrderBookStore {
  applySnapshot(symbol: string, snapshot: BookSnapshot): void
  applyUpdate(symbol: string, update: BookUpdate): void
  getTopOfBook(symbol: string): TopOfBook
  getSnapshot(symbol: string, depth: number): BookSnapshot
}

interface Registry {
  registerConsumer(handle: ConsumerHandle): void
  removeConsumer(id: string): void
  subscribe(consumerId: string, uri: ResourceURI): Subscription
  unsubscribe(consumerId: string, uri: ResourceURI): void
  onDemandChange(cb: DemandHandler): Unsubscribe
  status(): RegistryStatus
}
```

## Consideration Factors

- **Three seams (Bus / OrderBookStore / Registry)** — separates fanout (Bus) from book state (OrderBookStore) from consumer connection accounting (Registry). Each addresses a distinct concern; each has a different set of viable distributed implementations.
- **Single combined interface** — collapsing into one "DataPlane" interface obscures that the three concerns have different scaling and persistence properties (Bus is lossy fanout, OrderBookStore is current-state, Registry is connection accounting).
- **No seams (direct EventEmitter)** — fastest to write, but architecture cannot be lifted to split-tier without rewriting components. Fails the "extensible" requirement in DS-BRIEF.

## Rationale

DS-BRIEF asks for an architecture that "supports adding venues as a simple exercise" and is "structured to support future AI usage" — both of which require the components to depend on contracts, not implementations. Three seams instead of one makes the contracts honest about what each adapter actually has to do (a NATS Bus is very different from a Redis OrderBookStore is very different from a Redis Registry).

The seams are also what enables the three-variant deployment (DEC-016): in `MODE=gateway`, the binary literally cannot import the ingestion module's internals — they communicate only via these seams.

## Invalidation Trigger

Revisit if:
- The three seams collapse to one in practice (e.g., we adopt a single platform like Cloudflare Durable Objects that owns all three concerns).
- A scaling concern emerges that the current seams can't express cleanly (e.g., need durable replay; would need a fourth seam or a richer Bus interface).
