## Question

How do we simulate Coinbase's WebSocket behavior in deterministic CI tests? We need realism (so handlers actually exercise the right code paths) without the flakiness of hitting the live venue.

## Operator Decision

**Capture, then replay.** Build a small WebSocket server (`apps/coinbase-mock` or similar) that:

1. Replays from **fixtures captured from real Coinbase sessions** — the integration tests boot the mock with a recorded sequence of frames, including envelopes, sequence numbers, heartbeats, and timing.
2. Implements the Advanced Trade WS protocol surface our adapter uses: `subscribe` / `unsubscribe` ops on `level2` and `heartbeats` channels.
3. Exposes **fault-injection knobs** for tests:
   - Inject a sequence gap (skip a `sequence_num`).
   - Drop the connection mid-stream.
   - Stop emitting heartbeats (test the watchdog timeout).
   - Slow-emit messages (test backpressure paths).

Captures are recorded by a small companion script that connects to real Coinbase, records to disk for a few minutes, and stores the resulting fixture files in the repo. New captures can be regenerated when needed.

## Consideration Factors

- **Capture-and-replay (chosen)** — mock fidelity is grounded in observed reality, not protocol assumptions. The methodology operator articulated: *"mock from observed, not imagined."*
- **Pure synthetic mock generated from spec** — encodes our reading of the Coinbase docs, which may differ from actual server behavior in subtle ways (timing patterns, edge cases, undocumented fields). Tests pass against the mock but the system would fail against the real venue. The exact failure mode that "mock from observed" is meant to prevent.
- **Test against real Coinbase always** — flaky in CI (network, rate limits, market opening hours), can't deterministically inject failures, can't run in offline environments.
- **VCR-style HTTP fixture library** — Coinbase is WS-native; the WS-fixture libraries we'd need are less mature and less direct than just writing the small replay server ourselves.

## Rationale

The methodology this ADR fulfills is the take-home walkthrough's spine: "always-on first → observe → convert → **mock from observed** → automate." We've already run against real Coinbase by the time we build the mock (M1 dashboard requires live data; M2 demand-driven conversion observes lifecycle transitions against the real venue). The captures from those phases become the mock's source of truth.

Fault injection knobs are cheap to add at the replay layer. They give us deterministic tests for DEC-010's sequence-gap recovery and DEC-007's reconnect logic — handlers we've written defensively but couldn't test against the unpredictable real venue.

## Invalidation Trigger

- Coinbase changes their WS protocol substantially (e.g. Advanced Trade v3) — fixtures go stale; we re-capture and update. The mock structure stays.
- A third party publishes an officially maintained, faithful Coinbase mock — switch to it; less mock code we own.
- The mock becomes a maintenance burden disproportionate to the test value (e.g. spends most of its time keeping up with Coinbase changes). Could fall back to "tests against real Coinbase, manually triggered" as a degraded mode.
