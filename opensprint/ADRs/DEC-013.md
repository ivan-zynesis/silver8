## Question

MCP tools are request/response by default. How do we expose continuous market data updates to an LLM agent through MCP?

## Operator Decision

Use the protocol-native `resources/subscribe` mechanism. Each topic is exposed as an MCP resource at its `market://` URI; agents call `resources/subscribe` to register interest; the server emits `notifications/resources/updated` on each book change; agents re-read the resource (or receive pushed content depending on transport details) to get the current top-N snapshot.

Resource content is the *current view* (top-N levels as JSON), not raw deltas — diff application stays in the hub (DEC-009). The agent only ever sees a current snapshot.

## Consideration Factors

- **`resources/subscribe`** — protocol-native, matches MCP's intended streaming model, signals "AI-native" architectural intent. Requires a client that supports the subscribe primitive (Claude Desktop and the official MCP Inspector do; client support varies elsewhere).
- **Long-poll tool** (`poll_topic(topic, since_cursor, max_wait_ms)`) — works with every MCP client uniformly; degrades gracefully. But: less elegant; doesn't signal AI-native intent; LLM has to manage cursor state.
- **Snapshot-only + recommend WS for streaming** — punts on streaming through MCP entirely; agent has to switch protocols mid-task. Defeats the headline.

## Rationale

The eval criteria explicitly weight LLM usability and "structured to support future AI usage." Choosing the protocol-native streaming mechanism is the right architectural signal. It also forces us to think about the resource content shape, which is exactly the right thinking — "what does an LLM want to see when something changes?" answers to "current top-N snapshot," which feeds back into DEC-009.

We accept the constraint that some MCP clients have weaker `resources/subscribe` support; the dual transport (DEC-014) and the parity between MCP tools and the WS gateway (consumers can always fall back to WS) hedge this.

## Invalidation Trigger

Revisit if:
- Major MCP clients drop or never adopt `resources/subscribe` support; we'd need a long-poll fallback tool.
- The push semantics prove too noisy at high update rates (BTC-USD can update many times per second); might need server-side conflation or an explicit subscribe option for "snapshot every N ms."
