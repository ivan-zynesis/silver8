## Question

Which MCP transport(s) do we support, and what are the deployment implications?

## Operator Decision

Both:

- **HTTP+SSE**: primary for production-like use; agent connects to `https://hub.local/mcp` over the network. Lives alongside the WS gateway in the same container.
- **stdio**: supported via a `--transport=stdio` flag; agent spawns the process and pipes are wired by the MCP client. Useful for local testing (Claude Desktop integration, MCP Inspector via CLI).

The official `@modelcontextprotocol/sdk` is transport-agnostic at the server-logic level: same `Server` instance, swap `StdioServerTransport` for `SSEServerTransport`. Dual support is essentially free.

## Consideration Factors

- **HTTP+SSE only** — simplest deployment story. But local agent testing (Claude Desktop) is awkward — you'd need to run the server and configure HTTP transport, which not every client supports.
- **stdio only** — easiest local testing. But requires the agent to spawn the hub process — incompatible with running the hub as a long-running container with multiple consumers.
- **Both (chosen)** — single binary, two flags, no real cost. Local CLI testing via stdio; server-side deployment via HTTP+SSE.

## Rationale

DS-BRIEF says "MCP server can be deployed locally" — both transports satisfy this, but each addresses a different "local." HTTP+SSE is "container running on my machine"; stdio is "process spawned by my agent." Supporting both lets the operator and reviewers use whichever fits their workflow.

The architectural cost is minimal because the SDK isolates transport from server logic. Configuration: `MCP_TRANSPORT=http|stdio`; default `http`.

## Invalidation Trigger

Revisit if:
- A significant client class only supports a transport we don't implement.
- The SDK changes its transport abstraction in a way that makes dual support expensive.
