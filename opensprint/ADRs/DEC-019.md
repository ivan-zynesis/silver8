## Question

When a gateway pod is being terminated (autoscale-down or rolling deploy), how does it shed its consumers without dropping data on the floor?

## Operator Decision

A **protocol-level rebalance hint**: the server sends a "please reconnect when convenient" message to each consumer. Clients close gracefully and reconnect through the LB to a non-draining pod.

WebSocket gateway:
```json
{ "event": "rebalance", "reason": "scale-down", "deadline_ms": 30000 }
```

MCP server:
```json
notifications/rebalance { "reason": "scale-down", "deadline_ms": 30000 }
```

Drain sequence on `SIGTERM`:

1. `/readyz` flips to not-ready; LB stops sending new connections.
2. Broadcast rebalance hint to all current consumers.
3. Wait for `deadline_ms` (or until consumer count hits zero).
4. Close remaining connections cleanly.
5. Exit.

Combined with `terminationGracePeriodSeconds` (Kubernetes) or equivalent (60–90s typical).

## Consideration Factors

- **Server-initiated rebalance hint** — graceful, application-aware, protocol-supported. Right pattern for a system that already promises reconnect-friendly clients.
- **Wait for natural drain** — slow (minutes to hours); wastes capacity.
- **Force-close (RST or 1001)** — clients reconnect via LB and land elsewhere. Works because reconnect is already required, but is noisy and clients see errors.

## Rationale

The system already requires clients to handle disconnects (everything is reconnect-friendly per DS-COINBASE-WS analogy and DEC-011 lagged-disconnect). A rebalance hint formalizes "we'd like you to reconnect *now*, but cleanly" as a first-class protocol concept. It costs us a few lines of code and a documented notification — it gives the operator a real graceful drain.

Importantly, the rebalance event/notification is documented in `/docs` (DEC-023) as part of the failure-modes section — an LLM agent encountering it has a clear instruction: "close this connection and reconnect."

## Invalidation Trigger

Revisit if:
- Clients ignore the rebalance hint at scale (would need to fall back to force-close after deadline — already in the sequence).
- A class of consumer needs longer drain (e.g., long-running analysis); deadline becomes per-consumer or configurable.
