## Question

A consumer falls behind. What does the hub do?

## Operator Decision

Per-consumer-per-subscription bounded ring buffer with a **drop-oldest** policy. If the queue is full, the oldest message is evicted to make room for the newest. If the queue has been at-or-near capacity for a configured sustained-overflow window, the consumer is disconnected with a "lagged" reason code.

Implementation discipline:

- The outgoing message is **serialized once** per upstream tick, then the same buffer is `socket.send()`-ed to each subscribed consumer. Per-consumer queueing happens before serialization; per-consumer transport happens after.
- `socket.bufferedAmount` is checked against a watermark on every send. Exceeding the watermark counts toward sustained-overflow.
- Disconnect emits a clean close frame with reason; clients should reconnect (the protocol expects this).

Default thresholds (tunable via config):
- Queue depth: 1000 messages per subscription.
- Sustained-overflow window: 5 seconds.
- `bufferedAmount` watermark: 1 MB.

## Consideration Factors

- **Drop-oldest + sustained-overflow disconnect** — fresh data wins (correct for market data); incorrigibly slow consumers don't hold the system hostage.
- **Drop-newest** — wrong for market data; we'd hand out stale views indefinitely.
- **Block the publisher** — would propagate one slow consumer's badness to every consumer. Unacceptable for a fanout hub.
- **Conflate (collapse stale updates)** — sophisticated; useful for ticker; less obviously beneficial for L2 where each update is a discrete book change. Consider for future ticker topic.
- **Disconnect immediately on first overflow** — too aggressive; brief stalls are normal.

## Rationale

DS-BRIEF asks for an explicit, documented backpressure strategy. Drop-oldest + sustained-overflow is the canonical answer for streaming market data: prefer fresh over complete, and don't let one bad actor degrade the system.

Crucially, the drop is consumer-visible: the WS protocol includes a `lagged` event, and MCP includes a parallel notification, so an agent can detect that it has missed messages and react (typically: re-fetch the current snapshot via `get_book_snapshot`). This is documented in `/docs` failure modes per DEC-023.

## Invalidation Trigger

Revisit if:
- A specific consumer class has a strict no-loss requirement (e.g., regulatory replay) — they should not consume the lossy bus directly; they need a durable replay layer (Kafka territory, DEC-021).
- Sustained-overflow disconnects prove disruptive; could add a "recovering" backoff state where a consumer that drained quickly is given grace.
