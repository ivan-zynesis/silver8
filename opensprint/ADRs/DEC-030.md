## Question

Now that ingestion is demand-driven (DEC-027), an `OrderBookStore` holds nothing for symbols no consumer has yet asked for. So `/status`, MCP `list_topics()`, the dashboard's symbol selector, and the helpful-error pattern all lack a truthful answer to "what could a consumer ask for?" Where does that answer live, and who owns it?

## Operator Decision

Enrich the `VenueAdapter` (DEC-007) with **catalog capability**. Each venue adapter owns and exposes the catalog of topics it can serve. The catalog is conceptually distinct from active topics:

- **Catalog**: what could be subscribed to. Per-venue, owned by the adapter.
- **Active**: what is currently warm because of demand. Per-Registry, demand-driven.

```typescript
interface VenueAdapter {
  // ... existing members from DEC-007 ...
  listCatalog(): TopicDescriptor[]
  describeCatalogEntry(uri: ResourceURI): TopicDescriptor | undefined
  readonly catalogReady: boolean
  onCatalogChange?(cb: () => void): Unsubscribe
}
```

The catalog is **authoritative** for subscribe operations. WS gateway and MCP `resources/subscribe` reject subscribes to URIs not present in any adapter's catalog, returning the enumerated helpful-error pattern from DEC-015 (`unknown topic market://coinbase/book/BTC-USDT; available: market://coinbase/book/BTC-USD, …`). No lazy growth of catalog from successful subscribes.

Cross-venue policy (whether an aggregated catalog shows the union or intersection of venue catalogs; how to handle pairs available on one venue but not another) is **explicitly out of scope** for this initiative. There is one venue adapter today; the question does not yet arise.

## Consideration Factors

- **Enrich VenueAdapter (chosen)** — per-venue scope is natural; venue knowledge already lives in the adapter. A future Binance adapter exposes its own catalog without changes to `core`. Catalog readiness coheres with adapter readiness (DEC-033).
- **Standalone Catalog seam parallel to Bus / Store / Registry** — over-modeled. Catalog is inherently per-venue; a separate seam would either own one-per-venue catalogs (just adapter capability with extra ceremony) or a global aggregator (premature; cross-venue policy isn't decided).
- **Lazy growth from successful subscribes** — catalog is whatever someone has successfully asked for. Loses two properties: `list_topics()` stops being a complete answer to "what can I ask?", and helpful errors can't enumerate alternatives because the universe is unbounded. Defeats DS-LLM-USABILITY.
- **Keep `/status` of-active-only; punt on the cold-catalog problem** — leaves dashboards and cold agents with empty lists; architectural debt for a marginal scope saving.

## Rationale

The brief does not require this — a hand-coded list of common pairs in the dashboard's HTML would technically pass. We pay the marginal cost (a few interface members; a small refactor of `/status` and `list_topics`) for **architectural correctness**: the seam DEC-007 promised — that adding a venue is a simple exercise — is incomplete if every new venue must reinvent how the rest of the system discovers what it offers. Recording the rationale here makes the choice explicit, so a future maintainer asking "why did v1 ship this when the brief didn't mandate it?" finds a load-bearing answer rather than guessing.

The authoritative-catalog rejection of non-catalog subscribes is the corollary commitment. If catalog tells consumers what is askable, the system must enforce that promise — otherwise the catalog is decorative. Rejection follows the existing helpful-error pattern from DEC-015, so the LLM-facing experience is consistent with errors elsewhere in the surface.

## Invalidation Trigger

Revisit if:
- A consumer pattern emerges that genuinely needs lazy-grown topics (e.g. a research tool subscribing to arbitrary symbols for one-off backfill). Resolve by introducing a separate "lazy" topic kind, not by weakening the catalog-authoritative property of `book` topics.
- Cross-venue policy (ALL vs ANY) becomes load-bearing — that's a new ADR layered above this one, not a revision of this one.
- A platform abstraction emerges that owns both venue connection and catalog as one concept (Cloudflare Durable Object per venue, etc.) — the capability could move out of "adapter" into the platform, but the conceptual catalog/active split stays.
